# 规划任务 · HFQ Code **1.2** 大动 UI（+ F2 余量）

> **用法**：本文件是 **1.2 产品/设计规划契约**，不是立刻全量实现的 handoff。  
> **1.1.9 完成并发版后**，再拆 `1.2-*-handoff.md` 实现切片。  
> 用户排序（2026-07-21）：**先 1.1.9 小补丁 → 再 1.2 大动 UI**。  
> 基线：1.1.8 Layout A React 壳 + 更新 L3；编码体验债由 1.1.9 先清。  
> 落盘：`docs/prompts/1.2-ui-plan.md`

---

## 0. 一句话

**1.2 = 以 UI/UX 架构与视觉交互升级为主 train**，并择机并入 F2 余量（Memory 可检索、Goal 树产品化、panel prefs）；**不**默认重开更新管道或 IM 化。

---

## 1. 与 1.1.9 的边界

| | **1.1.9** | **1.2** |
|--|-----------|---------|
| 目标 | 编码闭环 **可用** | 工作台 **好用且像样** |
| 壳 / 布局 | **冻结** Layout A 信息架构 | **允许**大动视觉与面板编排（须书面选型） |
| Terminal/Changes/Tasks | 补行为洞 | 可重排信息架构与密度 |
| Memory / Goal OS | 不动 | **可**做 FTS + Goal 产品层 |
| 风险 | 低，可快发 | 高，必须分 slice + 设计门禁 |

**硬规则：1.1.9 未 ship 前，不合并 1.2 大改 UI 到 main 发版列车。**

## 2. 1.2 主题拆解

### 2.1 主轴 · UI 大动（必须）

在 **仍是「本地编码 Agent 工作台」** 前提下升级体验。候选工作包（实现前需产品勾选）：

| 包 | 内容 | 依赖 |
|----|------|------|
| **U0 · 设计门禁** | 更新 `LAYOUT-PROPOSALS` / 新 mock / 是否仍坚持 A 或引入 D 混合语法 | 人工选型 |
| **U1 · 设计系统** | tokens、密度、字体阶梯、surface/border、状态色；去「AI 紫模板感」 | U0 |
| **U2 · 壳层** | 活动栏 / 侧栏 / 顶栏 / 状态栏 / 抽屉行为与动画；窄屏折叠 | U1 |
| **U3 · Chat 工作台** | 时间线密度、工具卡、粘性 composer、空态、命令面板进化 | U1–U2 |
| **U4 · 编码三页** | Terminal / Changes / Tasks **版式**升级（1.1.9 已补行为） | U2 + 1.1.9 |
| **U5 · 次级页** | Settings / Skills / Memory / Models 统一 Page 语法 | U1 |

原则（继承）：

- 中文产品文案  
- **不**滑向 IM 网关  
- IPC 尽量稳定；大改优先 renderer  
- 反 big-bang：按 U0→U5 切片发版（可 1.2.0 / 1.2.1…）

### 2.2 副轴 · F2 余量（建议与 UI 分 slice，可同大版本）

| ID | 工作 | 说明 |
|----|------|------|
| F2-4 | Memory **FTS5 / 倒排** | 文件脑保留；搜索可用 |
| F2-Goal | **Goal OS 轻量产品化** | 多级树、状态、阻塞可见；**非**多租户 goals.db 照搬 |
| F2-5 | **Panel prefs** | 抽屉默认 tab、宽窄、是否记住 — 与 U2 强相关 |

Track E（embeddings / Python sidecar / MCP OAuth）**默认不进 1.2 主 train**。

### 2.3 明确不进 1.2 主叙事

- 默认开启静默更新（L3 保持 opt-in）  
- electron-updater 迁移（无书面例外）  
- ClawHub 发布端、IM、云多租户  
- 整库换 Python agent-core

## 3. 建议节奏

```text
[now]  1.1.9 polish handoff → 实现 → release:check → v1.1.9
          │
          ▼
[gate] U0 设计门禁（mock + 选型确认：A 深改 vs D 混合）
          │
          ▼
[1.2.0] U1–U3 设计系统 + 壳 + Chat（最大用户感知）
          │
          ├─► [1.2.x] U4–U5 编码三页 + 次级页统一
          │
          └─► [并行或紧随] F2-4 Memory FTS · Goal 轻量 · panel prefs
```

发版策略：

- **1.2.0** 可以是「看得见的 UI」为主，F2 可 1.2.1  
- 或 **一个 1.2 大列车** 多 commit，但仍按 U/F 切片验收

## 4. 设计门禁（U0 · 1.2 开工前必过）

1. 打开 `docs/design-proposals/` 与现行 Layout A 截图，写 **1 页选型纪要**（可附在 `LAYOUT-PROPOSALS.md` 或新 `docs/UI-1.2.md`）：  
   - 保留 A 骨架只换皮？还是引入 D 底栏终端等语法？  
   - 终端：继续右抽屉 vs 底栏可切换？  
   - 密度：IDE 紧 vs 阅读松？  
2. **人工确认** 前禁止大面积改 `AppShell` 信息架构。  
3. 确认后由统筹写 **`docs/prompts/1.2-ui-handoff.md`**（整包或 U1–U3 第一刀）。

---

## 5. 成功标准（1.2 列车结束时）

- 用户一眼感到 **工作台升级**（壳 + Chat + 密度），而非仅补丁号  
- 编码三页在新壳下仍闭环（依赖 1.1.9 行为底子）  
- Memory 至少 **可检索** 或书面说明延期 slice  
- `pnpm release:check` 绿；Windows 包可装；更新 L3 回归不坏  
- 文档：`RELEASE-1.2*.md` · ROADMAP baseline · DECISIONS 若改布局则记一笔
